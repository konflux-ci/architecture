# Trusted Tasks

Konflux allows users to extend their build pipelines with custom Tekton Tasks — for testing, scanning,
or other purposes. This extensibility introduces a question that Konflux must answer for every release:
*were the tasks that produced this artifact ones we actually trust?*

The answer comes from [Conforma] (previously called Enterprise Contract), which inspects the
pipeline run attestation generated by [Tekton Chains] and checks every task reference against
a set of trust data. If a task that touched critical parts of the build was not trusted at the
time of the run, the release is blocked.

This page describes how that trust model works: what "trusted" means, where trusted tasks come
from, how trust is established and eventually verified, and how [Trusted Artifacts] prevent
untrusted tasks from corrupting the critical build path even when they share a pipeline.

## What "Trusted" Means

A Tekton Task reference is trusted if Conforma can verify that it was released through a known,
controlled process. In practice, this means one of two things:

1. The task's OCI bundle digest (or git commit reference) appears on the `trusted_tasks` list
   published by Konflux, **and** it has not expired.
2. The task bundle URL matches an allow rule in `trusted_task_rules`, **and** no deny rule
   applies to it.

The full evaluation, in pseudocode from [ADR-0053]:

```text
(
  is_on_trusted_tasks_list(task_reference)
  or is_allowed_by_trusted_task_rules(task_reference)
)
and not expired_on_trusted_tasks_list(task_reference)
and not is_denied_by_trusted_task_rules(task_reference)
```

Denial takes priority. Even if a task is on the `trusted_tasks` list, a deny rule in
`trusted_task_rules` overrides it. The intent is to give policy authors a way to retire old
task versions across the fleet without waiting for individual teams to update their pipelines.

"Trusted" does not mean "verified to be correct" or "security-scanned." It means the task ran
through Konflux's own release pipeline and landed in a known registry location under conditions
that platform policy authors have accepted. The distinction matters: a trusted task can still
have bugs. What trust guarantees is *provenance* — where the task came from and that it passed
through the expected release gates.

## The Three Categories of Tasks

Tasks in Konflux pipelines fall into three categories based on where they come from and how
trust is established for each.

### Build Tasks

These are the tasks that make up a standard Konflux build pipeline: `git-clone`, `buildah`,
`prefetch-dependencies`, `clair-scan`, and similar. They live in
[`konflux-ci/build-definitions`](https://github.com/konflux-ci/build-definitions) and are
built and released by Konflux itself into `quay.io/konflux-ci/tekton-catalog`.

Build Service selects pipelines from this registry using the `build-pipeline-config` ConfigMap.
When a task bundle is released, its OCI digest is added to the
[`data-acceptable-bundles`](https://quay.io/repository/konflux-ci/tekton-catalog/data-acceptable-bundles)
artifact, which is the current source of truth for the `trusted_tasks` list.

As `build-definitions` decentralizes into multiple repositories (each team maintaining their
own task bundles), the `data-acceptable-bundles` update mechanism does not scale — parallel
releases from different repositories can corrupt the artifact. [ADR-0053] addresses this by
introducing `trusted_task_rules`, which lets a policy author trust all bundles released to
`quay.io/konflux-ci/tekton-catalog/*` by URL pattern rather than by explicit digest enumeration.

See [Trusted Build Tasks] for details.

### External (Third-Party) Tasks

Users and partners can include tasks that are not from `build-definitions` in their build
pipelines. Prior to [ADR-0053], this required the task to be submitted directly into
`build-definitions` and reviewed by the platform team ([ADR-0021], now replaced). Under the
decentralized model, a team can maintain their own task repository, release bundles to their
own registry, and configure `trusted_task_rules` to establish trust for those bundles.

External tasks that sit on the non-critical path of the build (i.e., they do not produce or
consume sources, compiled outputs, or provenance data) can be included safely because of
[Trusted Artifacts]. Tasks that do touch the critical path must themselves be trusted.

See [External Tasks] for details on how trust rules and Trusted Artifacts apply to third-party
tasks.

### Release Tasks

Release pipelines are a separate class entirely. They run in a privileged managed namespace,
not in the user's tenant namespace, and they are fetched via Tekton's git resolver from the
[`konflux-ci/release-service-catalog`](https://github.com/konflux-ci/release-service-catalog)
repository rather than from OCI bundles.

The release pipeline itself includes a Conforma task that gates whether the build being released
passes policy. This Conforma task is what checks the build pipeline's task trust — so the release
pipeline is the enforcement point, while the trust data (the `trusted_tasks` list or
`trusted_task_rules`) is the policy definition.

See [Trusted Release Tasks] for the trust model specific to release pipelines.

## How Trust Is Established

### The `trusted_tasks` List (Current)

The `data-acceptable-bundles` OCI artifact wraps a YAML file that maps each known task bundle
reference — by OCI URI and git URL — to a list of accepted digest/revision pairs, with optional
expiry dates:

```yaml
trusted_tasks:
  oci://quay.io/konflux-ci/tekton-catalog/task-buildah:0.4:
    - ref: sha256:9bedc54bbf756aee2cf9601892d1174e65dd6bc9a2cd350e7c9711acac4661e1
    - expires_on: "2025-11-16T00:00:00Z"
      ref: sha256:e44cc80f58cbf5a8c5130bcc5ab27efca87533e949dbbeb5541c3114e2610462
  git+https://github.com/konflux-ci/build-definitions.git//task/buildah/0.4/buildah.yaml:
    - ref: 4b23c64bd48914a6c4768b4b3e005ecc31e5b8d1
```

Each time a task bundle is released, the push pipeline updates this artifact with the new digest.
When the version tag in the registry moves to a newer release, the displaced digest begins
a 60-day expiry window before it is considered untrusted.

### The `trusted_task_rules` Mechanism (Emerging)

As task development decentralizes across repositories, the `data-acceptable-bundles` update
mechanism becomes a bottleneck. [ADR-0053] introduced `trusted_task_rules`, a rule-based
alternative that will operate alongside `trusted_tasks` until the transition is complete.

Rules can allow all bundles matching a URL pattern, require a signing key to be present, and
expire specific version ranges on a schedule:

```yaml
trusted_task_rules:
  allow:
    - name: Trust all tasks from konflux-ci/tekton-catalog
      pattern: oci://quay.io/konflux-ci/tekton-catalog/*

    - name: Require common signing key (from 2026 onwards)
      pattern: oci://quay.io/konflux-ci/tekton-catalog/*
      signing_key: <common public key>
      effective_on: 2026-01-01
  deny:
    - name: Retire buildah versions below 0.5
      pattern: oci://quay.io/konflux-ci/tekton-catalog/task-buildah*
      versions:
        - '<0.5'
      effective_on: 2025-11-15
```

The `effective_on` field is critical. It lets policy authors announce a deprecation in advance
and give users time to update, rather than failing builds immediately. Conforma evaluates rules
with `effective_on` only after that date has passed.

### Task Signing

Trusting all bundles at `quay.io/konflux-ci/tekton-catalog/*` by URL alone is not a
fundamentally stronger trust model than `data-acceptable-bundles` — anyone who can write to that
namespace can publish a bundle. The long-term plan from [ADR-0053] is to sign all upstream
Konflux task bundles with a common release key. Once signing is in place, policy authors can
require that bundles pass signature verification, which limits trust to builds that ran through
the Red Hat deployment of Konflux's own release pipeline.

### Task Versioning and Expiry

The `trusted_tasks` expiry mechanism depends on tasks being released repeatedly under the same
version tag, so that the "old" digest can be identified when the tag moves forward. This breaks
down when tasks are developed and versioned across many repositories independently.

[ADR-0054] addresses this by requiring each meaningful change to a task to carry a version
bump in the `app.kubernetes.io/version` label, and for that version to be recorded in the
`org.opencontainers.image.version` OCI annotation on the released bundle. With proper semantic
versions on bundles, `trusted_task_rules` can deny specific version ranges:

```yaml
deny:
  - name: Expire buildah below 0.5.1
    pattern: oci://quay.io/konflux-ci/tekton-catalog/task-buildah*
    versions:
      - '<0.5.1'
    effective_on: 2025-11-29
```

This replaces the implicit expiry that relied on the version tag moving forward. Task expiry
can also be set manually on any individual task bundle using the
`build.appstudio.redhat.com/expires-on` annotation; Conforma will honour that regardless of
which trust mechanism allowed the task in the first place.

## Role of Trusted Artifacts

[Trusted Artifacts] (implemented in [ADR-0036], mandated for all component build pipelines by
[ADR-0063]) are a complementary but separate mechanism. They solve a different problem:
how can a build pipeline include user-provided or external tasks without those tasks being
able to tamper with the build outputs that flow between trusted tasks?

The answer is that critical files — source code, prefetched dependencies, build context — are
never passed through Tekton workspaces backed by Persistent Volume Claims. Instead, they are
wrapped into archives, stored in an OCI registry, and their digests are recorded as Tekton task
results. When a consuming task receives one of these archives, it verifies the checksum before
unpacking. An untrusted task in the middle of the pipeline cannot modify the archive without
breaking the checksum.

This means the trust model and Trusted Artifacts work in different layers:

- **Trusted Artifacts** protect data integrity on the critical path.
- **Trusted Tasks** protect provenance — they ensure that the tasks which produced the build
  output were ones Conforma knows about.

Both are checked by Conforma at release time. A build pipeline that uses Trusted Artifacts
but runs an untrusted `buildah` task will still fail Conforma's policy check. A pipeline
whose tasks are all trusted but that bypasses Trusted Artifacts (sharing outputs through
a generic workspace) will also fail.

## Relationship with Conforma

[Conforma] performs all trust verification. It does not run inline during the build — it runs
as a Tekton task inside the release pipeline, after the build has completed, reading the
pipeline run attestation that [Tekton Chains] produced.

The attestation records every task that ran, the parameters each task received, and the results
each task produced. Conforma reads this attestation and evaluates the trust rules against each
task reference. If any task that is considered to be on the "critical path" of the build (as
declared in the pipeline's Conforma policy) was not trusted, the release is blocked.

The [Enterprise Contract](./enterprise-contract.md) documentation describes how to configure
an `EnterpriseContractPolicy` CRD and what policy sources Conforma evaluates. The trusted task
rules themselves live in the policy data sources, not in the ECP CRD.
